# 03 · 收成节流算法（核心）

> **更新（2026-06-14）**：`T_MIN` 由 0.5s 改为 **1.0s**（最快 1 秒丰收一次）。
> 另：资产动画分 **4 个抽象阶段** working1(<25%)/working2(<50%)/working3(<75%+)/harvest(满)，
> 由引擎按进度发 `onEvent(event, phaseSec)`，`phaseSec` 为该阶段动态时长（=周期 D 的相应比例），
> 皮肤据此把动画铺满该阶段——**无固定时长**，周期快则演得快。下文 0.5 处按 1.0 理解。

这是整个产品体验的数学核心：**如何把"匀速增长的资产"转化为一条会满的进度条
和一个个 `+X` 的弹出**，同时满足两条硬约束。

## 1. 需求约束（来自产品定义）

设某资产每秒产出速率为 `r`（展示单位/秒，恒为正）。

- **C1（频率上限）**：进度条"满"的频率**最快 0.5 秒一次**。即两次收成之间间隔 ≥ `T_MIN = 0.5s`。
- **C2（金额下限）**：每次弹出的 `X` 为**整数且 ≥ 1**。
- **C3（行为期望）**：
  - 产出快（`r` 大）→ 每 ~0.5s 弹一个 `X > 1` 的数。
  - 产出慢（`r` 小）→ 攒够 1 再弹（间隔自然 > 0.5s），`X = 1`。
- **C4（不漂移，隐含）**：长期来看弹出总额应等于真实产出，不能凭空多给或少给。

## 2. 核心公式

对每个资产，定义一个"收成周期"：

```
T_MIN = 0.5                          // 秒，频率上限
X = max(1, ceil(r * T_MIN))          // 本周期弹出的整数金额
D = X / r                            // 本周期时长（秒）
```

进度条在 `D` 秒内从 0 填到满；满时弹出 `+X`，账本累加 `X`，进度条归零，重复。

### 为什么这套公式同时满足全部约束

- **满足 C2**：`X = max(1, ...)` 保证 `X ≥ 1` 且为整数（`ceil` 取整）。
- **满足 C1**：因为 `X = ceil(r·T_MIN) ≥ r·T_MIN`，所以
  `D = X/r ≥ (r·T_MIN)/r = T_MIN = 0.5s`。**周期永远 ≥ 0.5 秒**，频率不会超上限。
- **满足 C3**：
  - 快（`r·0.5 ≥ 1`）：`X = ceil(r·0.5) > 1`，`D ≈ 0.5s` → 每半秒弹一个大数。✅
  - 慢（`r·0.5 < 1`）：`X = 1`，`D = 1/r > 0.5s` → 攒够 1 再弹，间隔拉长。✅
- **满足 C4（关键）**：弹出速率 = `X / D = X / (X/r) = r`。
  **长期产出速率精确等于真实速率 `r`，零漂移、无需 carry-over 余数。** ✅

> 这套设计的优雅之处：`X` 与 `D` 由同一个 `r` 推出，天然自洽——
> 不需要额外维护"已攒未发"的浮点余数，避免累积误差。

## 3. 数值示例

设展示单位为元，`SECONDS_PER_YEAR = 31,536,000`。

| 场景 | 年产出 | r（/秒） | r·0.5 | X=ceil(r·0.5) | D=X/r | 表现 |
|---|---|---|---|---|---|---|
| 小额工资 | 120,000 | 0.00380 | 0.0019 | 1 | ~263s | 每 ~4.4 分钟 +1 |
| 中等投资 8% | 1,000,000×0.08=80,000 | 0.00254 | 0.0013 | 1 | ~394s | 每 ~6.6 分钟 +1 |
| 大额组合 | 60,000,000 | 1.902 | 0.951 | 1 | ~0.53s | 每 ~0.53s +1 |
| 巨额 | 200,000,000 | 6.34 | 3.17 | 4 | ~0.63s | 每 ~0.63s +4 |
| 超巨额 | 1,000,000,000 | 31.7 | 15.8 | 16 | ~0.50s | 每 ~0.5s +16 |

> 说明：真实"小额"资产攒 1 元要几分钟——进度条**缓慢爬行**正是"细水长流"的治愈感来源；
> 想要更频繁的正反馈，可在隐私模式下把 `goldRate` 设小（1 金币=很少的钱），
> 让金币产出更快、进度条更活跃。这是给用户的一个"体验旋钮"。

## 4. 隐私模式 / 金币下的表现

展示单位为金币时，`r` 用金币速率：`r_gold = r_currency / goldRate`。
公式不变，`X` 即金币整数，进度条节奏由 `goldRate` 调节（见上 note）。

## 5. 实现细节

### 5.1 状态（每个资产一份）

```ts
interface HarvestState {
  r: number;          // 每秒速率（展示单位），随资产/模式变化时重算
  X: number;          // 当前周期目标金额（整数）
  D: number;          // 当前周期时长（秒）
  elapsed: number;    // 本周期已过时间（秒）
}

function initHarvest(r: number): HarvestState {
  const T_MIN = 0.5;
  const X = Math.max(1, Math.ceil(r * T_MIN));
  return { r, X, D: X / r, elapsed: 0 };
}
```

### 5.2 每帧更新（由渲染循环 / requestAnimationFrame 驱动）

```ts
// dt: 距上一帧秒数
function tickHarvest(s: HarvestState, dt: number, onHarvest: (x: number) => void) {
  s.elapsed += dt;

  // 可能一帧跨过多个周期（卡顿/后台标签页恢复时）——用 while 补发
  while (s.elapsed >= s.D) {
    s.elapsed -= s.D;
    onHarvest(s.X);           // 弹 +X、播放收成动画、账本累加
  }
}

// 进度条填充比例（给渲染层）
const progress = s.elapsed / s.D;   // 0..1
```

### 5.3 后台/离线时间的处理 ✅（已定：补发离线收益）

**决策：离线挂机收益必须有**——资产"一直在为你工作"是核心理念。

实现：
- 持久化每个资产的"上次结算时间戳"（或全局一个）。重新打开 / 挂件回前台时，
  用真实流逝时间 `Δ = now - lastSettled` 补算应得收成。
- 计算用闭式公式而非真跑 `while`（离线可能几天，循环会很慢）：
  `离线收成 ≈ floor(r × Δ)`（展示单位整数），一次性入账。
- **合并展示**：超过阈值（如离线收成 > 若干次或 > 几秒动画量）时，
  不逐个播放动画，而是弹一次「离线收成 +N 🌙」汇总，再恢复正常实时节奏。
- 前台实时阶段仍用 §5.2 的 `while` 补发（处理短暂卡顿/标签页节流）。

### 5.4 多资产

- 每个资产独立维护 `HarvestState`，独立进度条与动画。
- 全局只有一个 rAF 主循环，遍历所有资产 `tickHarvest`，收成回调统一写账本。
- 性能：资产数量在 MVP 量级（几十个）完全无压力；PixiJS 负责渲染合批。

### 5.5 速率变化（编辑资产 / 切换模式）

- 用户编辑金额、切换隐私模式 → 重算 `r` → `initHarvest(r)` 重置当前周期。
- 简化：重置会丢弃当前周期已 `elapsed` 的进度（视觉上进度条回弹到 0）。
  MVP 可接受；若想平滑，可按比例迁移 `elapsed`，非必须。

## 6. 测试要点（TDD）

- `initHarvest`：各 `r` 下 `X≥1`、`D≥0.5`、`X` 为整数。
- 不漂移：模拟运行 `N` 秒，累计弹出 ≈ `r·N`（误差 < 一个 `X`）。
- 频率上限：任意 `r` 下相邻收成间隔 ≥ 0.5s。
- 跨周期补发：大 `dt` 时 `while` 正确补发多次。
- 边界：`r` 极小（趋近 0）、极大；`goldRate` 改变后的金币速率。
